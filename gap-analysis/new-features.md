# New Features / Enhancements Backlog

**Last updated:** 2026-03-01

Priority uses MoSCoW: Must, Should, Could, Won't (this release).

## Feature Table

| ID | Feature Idea | User Problem + Persona | Value Hypothesis | Scope | Data/Schema Impacts | Security/Privacy | Telemetry | Priority |
|---|---|---|---|---|---|---|---|---|
| FEAT-001 | Unit test suite for all internal packages | **Engineers** cannot safely refactor or extend internal packages; any change risks undetected regressions | Unit tests reduce bug introduction by 40-80% and enable confident refactoring; currently 33/34 packages have zero unit tests | MVP: auth, secrets, tools, session, state, policy. Later: remaining packages | None | Tests for security packages (auth, secrets, policy) directly improve security posture | Test coverage metrics per package | Must |
| FEAT-002 | CI security scanning (govulncheck + gosec) | **DevOps/Security** have no automated detection of known CVEs or insecure code patterns | Catches vulnerable dependencies and insecure patterns before merge; industry standard practice | MVP: Add govulncheck + gosec to CI workflow | None | Direct security improvement | CI job pass/fail metrics | Must |
| FEAT-003 | CI test coverage reporting and thresholds | **Engineers** have no visibility into which code paths are tested or if coverage regresses | Coverage gates prevent regression; visibility drives test writing culture | MVP: coverprofile + Codecov upload + 70% threshold | None | None | Coverage percentage trend | Must |
| FEAT-004 | CI race detection job | **Engineers** have shared-state code (maps, slices) with potential races that are undetected | Race detector catches concurrency bugs that are otherwise intermittent and hard to reproduce | MVP: Add `go test -race` CI job | None | Races in auth/session code are security-relevant | Race detection pass/fail | Must |
| FEAT-005 | RBAC / multi-user access control | **Platform admins** deploying shared agent servers need per-user/per-agent access control | Single API key is insufficient for teams; no audit trail of who did what | Later: user model, role definitions, per-agent permissions, audit log | New: users table, roles table, user_agent_permissions | Major: auth system redesign, session binding to users | Per-user invocation metrics, access denied events | Should |
| FEAT-006 | Agent observability dashboard | **SRE/Platform engineers** cannot monitor agent health, latency, error rates, or token usage in real-time | Agents are black boxes in production; no alerting on degradation or cost overruns | MVP: Grafana dashboard template. Later: built-in web dashboard | None (uses existing telemetry data) | Dashboard auth | Agent latency p50/p95/p99, error rate, token usage, tool call frequency | Should |
| FEAT-007 | Agent versioning and rollback | **Engineers** deploying agents in production need safe rollbacks when a new version causes issues | No rollback mechanism; re-applying old `.ias` file requires manual diff to verify state | MVP: version history in state file + `agentspec rollback` command. Later: blue-green deployment | State file: add version history array, max 10 versions | Version history may contain secrets if agent config includes inline secrets | Rollback frequency, version drift | Should |
| FEAT-008 | Interactive agent testing / playground | **Engineers** building agents want to test prompts and tool configs without deploying | Currently must `apply` + `run` to test; slow iteration loop | MVP: `agentspec test` command with mock tools and recorded responses. Later: web-based playground | None | Test fixtures should not contain real API keys | Test execution time, pass/fail rate | Should |
| FEAT-009 | Multi-model support and fallback chains | **Engineers** want agents that fall back to cheaper models when primary is unavailable or for non-critical tasks | Currently one model per agent; no cost optimization or resilience via model switching | MVP: model priority list in agent config; fallback on error/timeout. Later: routing by task complexity | Agent IR: add `models` list with priority, fallback config | API keys for multiple providers | Model selection frequency, fallback rate, cost per model | Should |
| FEAT-010 | Structured logging with correlation IDs | **SRE/Platform engineers** cannot trace a request across tool calls, LLM invocations, and session operations | Log entries are disconnected; impossible to reconstruct request flow for debugging | MVP: Request-scoped correlation ID in all logs. Later: distributed tracing with OpenTelemetry | None | Correlation IDs in logs help audit trails | Log volume, trace completeness | Should |
| FEAT-011 | Agent guardrails / content filtering | **Platform admins** need to prevent agents from producing harmful, off-topic, or policy-violating output | No output filtering; agents can produce any content the LLM generates | MVP: regex-based output filters + keyword blocklist. Later: LLM-based content classifier | Agent config: add `guardrails` block with filter rules | Critical: prevents harmful content generation | Filter trigger frequency, blocked response rate | Should |
| FEAT-012 | Hot-reload without server restart | **Engineers** in dev mode want instant feedback when changing `.ias` files | Current dev mode has 2-second polling latency and restarts the entire server | MVP: fsnotify-based watching + graceful config reload without dropping connections | None | None | Reload latency, reload failure rate | Could |
| FEAT-013 | Plugin marketplace / registry | **Engineers** want to discover, share, and install community-built plugins | Currently plugins must be manually compiled and placed in the correct directory | MVP: `agentspec plugin search/install` with a public registry. Later: verified publishers, ratings | New: plugin registry API, plugin metadata schema | Plugin supply chain security: signing, verification, vulnerability scanning | Plugin install count, registry search queries | Could |
| FEAT-014 | Conversation export and replay | **Engineers** want to export conversation histories for debugging, compliance, or training data | Conversations are ephemeral (memory store) or stored in Redis without export capability | MVP: `agentspec sessions export --format jsonl/csv`. Later: replay command for debugging | None (reads existing session data) | PII in conversations must be considered; export should support redaction | Export frequency, export size | Could |
| FEAT-015 | Cost tracking and budgets | **Engineering managers** need visibility into LLM API costs per agent, session, and time period | Token counts are tracked in metrics but not translated to dollar costs or budget limits | MVP: Cost estimation based on token counts and model pricing. Later: hard budget limits that pause agents | Agent config: add `budget` block with daily/monthly limits | Budget data may be business-sensitive | Cost per agent, budget utilization, budget exceeded events | Could |
| FEAT-016 | Integration test recording and replay | **Engineers** want deterministic integration tests that don't require live LLM APIs | Current mock client is manually configured; no way to record and replay real interactions | MVP: HTTP recording middleware for LLM calls. Later: auto-fixture generation | None | Recorded fixtures must not contain API keys | Fixture freshness, replay fidelity | Could |
| FEAT-017 | Agent-to-agent communication protocol | **Engineers** building multi-agent systems need structured inter-agent communication beyond simple delegation | Current delegation is fire-and-forget string passing; no structured messaging, no pub/sub | MVP: Typed message passing with schemas. Later: event-driven pub/sub between agents | Agent IR: add `messages` block with input/output schemas | Inter-agent data flows may cross trust boundaries | Message volume, delegation latency, cross-agent error rate | Could |
| FEAT-018 | OpenTelemetry native integration | **Platform engineers** want standard observability that integrates with existing monitoring stacks | Current telemetry is custom Prometheus metrics; not compatible with OTLP ecosystem | MVP: OpenTelemetry SDK for traces and metrics. Later: auto-instrumentation | None | Trace data may contain sensitive request content | Trace export success rate, metric cardinality | Could |
